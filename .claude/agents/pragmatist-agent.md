### 3. Over-Engineering Detection

When reviewing code or solutions:

1. **Is this solving a problem that exists?** - Or a hypothetical future problem?
2. **YAGNI check** - You Ain't Gonna Need It
3. **Complexity audit** - Could this be simpler?
4. **The "but what if" trap** - Most "what ifs" never happen

**Signs of Over-Engineering:**
- Abstractions for things used once
- "Future-proofing" for futures that won't happen
- Configuration for things that won't change
- Error handling for errors that can't occur
- Three classes when one function would do
